## 这周到底发生了什么

百度 ERNIE 团队在 2026-06-22 发布、6-25 在 GitHub 与 Hugging Face 同步开源 **Unlimited OCR**——一款专门面向「长文档端到端识别」的开源 OCR 多模态模型。

- 模型名：**baidu/Unlimited-OCR**
- 仓库：**https://github.com/baidu/Unlimited-OCR**
- 镜像：https://huggingface.co/baidu/Unlimited-OCR
- 论文：arXiv 2606.23050

不到一周，它做到了三件事：**GitHub Daily Trending 登顶、5 天内 Star 数突破 1 万、OmniDocBench v1.6 综合得分把 DeepSeek OCR 拉下榜首**。在过去的国产开源 OCR 赛道里，这是第一次有一个非「大厂全家桶」的轻量模型，把端到端基准跑到 SOTA。

## 数字盘点

下面这组数字是 Unlimited OCR 这一次破圈的「现场温度计」：

- **3B**：模型总参数量。比同类多模态 OCR 旗舰小一个数量级。
- **570M**：推理时实际激活的参数量。轻量级推理路径，单张消费级 GPU 就能跑。
- **1 万 Star / 5 天**：开源到登上 GitHub Daily 榜首再到突破 1 万 Star 的全过程总用时。
- **93.92%**：OmniDocBench v1.6 综合得分（端到端 SOTA）。v1.5 是 93.23%。Distinct-35 子项跑到 **96.90%**。
- **0.1069**：编辑距离指标（越低越好）。
- **+12.7%**：在真实文档场景下，相对 DeepSeek OCR 的速度优势。
- **+35%**：在 6000 tokens 长输出场景下，相对 DeepSeek OCR 的速度优势。
- **128 tokens**：R-SWA 创新点中 KV Cache 的恒定窗口大小。

## R-SWA 到底解决了什么问题

要看懂 Unlimited OCR 的意义，必须先看一个细节——它的核心创新叫 **R-SWA（Reference Sliding Window Attention）**。

传统 OCR 多模态模型有一个老毛病：当输入是一份长文档（几十页 PDF、含表格、含手写批注）时，模型要在生成几千个输出 token 的过程中反复回看图像内容，KV Cache 像滚雪球一样越滚越大，显存与算力一起飙升。**结果是模型越长、越慢、越贵——本质上长文档场景反而是「最贵的 OCR 调用」**。

R-SWA 给出的解法非常优雅：

- 维持一个**恒定 128 token 的滑动窗口**，让模型每次只看「最近最关键的视觉锚点」；
- 把整篇文档的「视觉参考」结构化成多个 reference 点，模型在窗口内逐点定位、按需回看；
- **KV Cache 不再随输出长度增长**——意味着不管你要识别 1 页还是 100 页，单 token 的推理成本几乎是常数。

这就解释了为什么 6000 tokens 的长输出场景下，Unlimited OCR 比 DeepSeek OCR 快 35%——**对手是「随长度变慢」，它是「不随长度变慢」**。这不是小修小补，这是 OCR 推理架构的小型范式跳跃。

## 为什么 3B 模型能干赢更大的对手

OCR 这个赛道有个反直觉的规律——**更大的模型不一定更准，但一定更慢更贵**。原因在于 OCR 的本质是「视觉 → 结构化文本」，需要的「世界知识」远比「视觉理解 + 排版重建」要少。把通用大模型直接拿来跑 OCR，相当于用一辆 V12 SUV 在小区里送快递——浪费、慢、堵车。

百度 Unlimited OCR 的设计哲学正好相反：

- **参数定位轻量化**：总参 3B、激活 570M——单张 4090 / A10 就能跑通；
- **数据定位场景化**：训练数据高度集中在文档、票据、手写、表格、公式这五个 OCR 真实场景；
- **架构定位推理友好**：R-SWA 让长输出场景的 KV Cache 不爆炸，配合稠密 KV 共享降低显存压力。

合起来的效果：**OmniDocBench v1.6 综合 93.92% SOTA + 比 DeepSeek OCR 快 12.7%–35% + 单卡可部署**。这三件事捆在一起，开发者社区的反应就只剩一个动作——立刻把它克隆下来跑 demo。5 天 1 万 Star 不是营销，是真实的「试过都觉得值」。

## 业内为什么把它和 DeepSeek 拿来对比

DeepSeek OCR 是过去半年开源 OCR 赛道的明星。它从 LLM 大模型脉络里抽出 OCR 子能力，做出了一款被广泛部署的开源工具。这一次百度的 Unlimited OCR 把它从端到端 SOTA 的位置上拉了下来——业内反应非常微妙。

有趣的是，Unlimited OCR 的论文作者署名里，**疑似有前 DeepSeek 研究员的身影**（魏浩然）。这件事一旦坐实，就是一个非常有故事性的产业转身：「<b>前 DeepSeek 团队的人，到百度做了一款把 DeepSeek OCR 反超的产品</b>」。当然，这只是社区八卦层面的故事，真正决定胜负的还是技术与产品。

但故事的「叙事力」很强——它给国内开源 OCR 赛道注入了一种「内部竞合 + 人才流动」的氛围。**人才在大厂与初创之间反向流动，产品在开源仓库里互相切磋，受益方是所有把 OCR 接入到自己工作流的开发者**。

## 这件事对开源生态的意义

开源 OCR 听起来像一个很「工具向」的细分赛道，但它在 2026 年的产业链里，是一条隐性的核心基础设施。

- **企业知识库构建**：90% 的企业数据躺在 PDF/扫描件/截图里，OCR 是把它喂给大模型的第一道门；
- **Agent 真实可用性**：让 Agent 真正能「读屏」「读单据」「读发票」，OCR 是必备的眼睛；
- **多模态训练数据扩展**：开源高质量 OCR 越多，多模态预训练数据的可获得性就越好。

百度这次开源选择得很「狠」——直接放出 3B 的训练好模型 + Hugging Face 镜像 + GitHub 仓库。这种「大厂全开源」的姿态，恰好和 LongCat-2.0、Qwen3.5、DeepSeek V4 等近期的国产模型开源大潮节奏对上：**国产 AI 的开源生态，正在从「单品开源」走向「全栈开源」**。

## 几个值得开发者关注的细节

**① 部署门槛**：单张消费级 GPU（4090/A10 级别）就能跑起完整推理，团队无需采购 H100。

**② 商用授权**：仓库挂出的是**商用友好**协议，企业可以把它接进生产环境而不用额外谈判。

**③ R-SWA 范式可外推**：R-SWA 的"恒定窗口 KV Cache"思路，不仅适合 OCR，理论上也适合任何"长输出 + 强参考"的多模态场景（如表格生成、报告生成、字幕生成）。这是 Unlimited OCR 留给行业的最重要技术礼物。

**④ 评测口径完整**：团队在论文里同步公布了 OmniDocBench v1.5 / v1.6 / Distinct-35 / 编辑距离四套口径——这种"不挑分数"的诚意，是开源社区最值钱的信誉。

## 留给普通读者的三条朴素观察

**① 别把"OCR"想成传统 OCR。** 2026 年的 OCR 已经不只是"识别字"，而是"把文档变成可被 AI 直接调用的结构化信息"。一份扫描合同进去，出来的不只是文字，是带标签的字段、表格、签字位置。
